# نَظْم

**نَظْم** هو مُجمِّع عربي لبنية **x86-64** مكتوب بلغة C11. هدفه تحويل ملفات تجميع عربية بامتداد `.مجمع` إلى ملفات كائنية قياسية يمكن ربطها بأدوات النظام مثل `ld` على لينكس أو أدوات COFF على ويندوز.

المشروع في مرحلة مبكرة `0.3.0`: توجد بنية خط أنابيب واضحة، ووحدات أساسية للترميز، والرموز، والمحلل المعجمي، وجدول الرموز، ومخرجات ELF64/COFF. يدعم الإصدار الحالي أقسام `.text` و`.data` وسلاسل UTF-8 العربية وrelocations أساسية لمراجع الوسوم داخل `.text`، مع بقاء التحقق النهائي ضد أدوات الربط على ويندوز ضمن خارطة الطريق.

الهدف الاستراتيجي هو أن يصبح نَظْم المُجمِّع الإنتاجي لمترجم **باء**:
يستبدل خرج AT&T/GAS الإنجليزي الحالي بمصدر تجميع عربي قابل للفحص، ثم ينتج
ملفات ELF64/COFF قبل تمريرها إلى رابط النظام. وعلى المدى البعيد، بعد تثبيت
العقد واختبارات المطابقة، يُعاد تنفيذ نَظْم بلغة باء ويُبنى عبر مسار bootstrap
يحافظ على تنفيذ C11 الحالي كمرحلة صفر موثوقة.

> **تنبيه الحالة:** نَظْم ليس جاهزا بعد ليحل محل GAS في بناء باء الإنتاجي.
> أُغلقت بوابة صحة الترميز الحالية 7.1، لكن ما زالت توجد فجوات في الرموز
> الخارجية وrelocations وتغطية تعليمات باء والتحقق الفعلي عبر روابط لينكس
> وويندوز.

## الفكرة

نَظْم يحاول جعل البرمجة منخفضة المستوى أقرب للناطقين بالعربية دون التخلي عن صرامة تنسيق الملفات الكائنية أو قواعد x86-64. يقرأ المصدر العربي كـ UTF-8، يتعامل مع التعليمات والسجلات والوسوم العربية، ثم يمررها عبر خط أنابيب متعدد المراحل:

```text
مصدر .مجمع
  -> التحليل المعجمي
  -> التحليل النحوي
  -> المرور الأول: أحجام التعليمات والوسوم
  -> المرور الثاني: ترميز x86-64
  -> كاتب المخرجات: ELF64 أو COFF
  -> ملف .o أو .obj
```

## الحالة الحالية

- الإصدار الحالي في `CMakeLists.txt` وواجهة `include/nazm.h` وأمر `--version` هو `0.3.0`؛ ويعرض الأمر أيضا هدف بناء المشغل مثل `x86_64-linux` أو `x86_64-windows`.
- اللغة الأساسية للمشروع هي C11 بدون اعتماديات تشغيل خارجية.
- الاختبارات الحالية تغطي وحدات الذاكرة، Unicode العربي، جدول الرموز، الكلمات المفتاحية، الفوريّات، بادئة REX، المحلل المعجمي، المحلل النحوي، الترميز، المرورين، ELF64، COFF، التشخيصات، أمثلة المصدر، وتحليل خيارات CLI.
- يرفض مشغل CLI ملفات المصدر الأكبر من 100 MiB برسالة عربية واضحة بدلا من محاولة تحميلها كاملة.
- تعرض التشخيصات العربية الآن موضع الخطأ مع السطر الأصلي وسهم يحدد امتداد الرمز أو المعامل عند توفر المصدر.
- يرفض parser إزاحات الذاكرة خارج signed 32-bit، ويرفض المرور الثاني أي قفز
  أو نداء نسبي لا تمثله `rel32` بدلا من اقتطاع القيمة.
- يطابق الإصدار 0.3 الكلمات المفتاحية والوسوم كبايتات UTF-8 دون تطبيع أو
  aliases ضمنية؛ الحركات المكتوبة في الصيغة canonical مطلوبة. راجع
  [سياسة Unicode](Docs/UNICODE.md).
- الوسوم محلية افتراضيا؛ يصدّر `.عام اسم_الوسم` الرمز للرابط، ويثبّت
  `.محلي اسم_الوسم` الرؤية المحلية صراحة. يقبل التصريح قبل تعريف الوسم أو
  بعده، ويرفض التعارض أو بقاء الوسم بلا تعريف.
- توجد اختبارات أمثلة تمر عبر خط الأنابيب المكتبي لكل ملفات `examples/*.مجمع` الجيدة وتنتج ELF64 وCOFF في الذاكرة.
- إخراج ELF64 وCOFF يدعم `.text` و`.data` وجداول الرموز وrelocation أساسية من `.text` إلى الوسوم، لكن الربط النهائي عبر أدوات النظام المختلفة ما زال يحتاج تحقق CI ومنصات فعلية.
- التكامل مع باء مخطط على مراحل: جرد خرج Machine IR الحالي، ثم إضافة خيار
  `gas|nazm` اختياري للمقارنة، ثم جعل نَظْم افتراضيا بعد نجاح اختبارات باء على
  لينكس وويندوز. التفاصيل في
  [Docs/BAA_INTEGRATION.md](Docs/BAA_INTEGRATION.md).

## المتطلبات

- CMake 3.20 أو أحدث.
- مترجم C يدعم C11، مثل GCC 11+ أو Clang 14+.
- على لينكس: `make` ويفضل توفر `ld` أو `lld` للتجارب النهائية.
- على ويندوز: MinGW أو MSVC، مع مراعاة أن دعم COFF يحتاج تحقق مستمر.

## البناء

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build -j
```

أو عبر المسار المباشر الخفيف دون CMake:

```bash
./build.sh
```

## الاختبار

```bash
./build.sh test
```

يشغل المسار المباشر حاليا 17 مجموعة اختبارات. تشغّل CTest المجموعات نفسها،
وتضيف مقارنة بايتات تفاضلية مع GNU `as` و`objcopy` عندما تتوفر الأداتان.

وللتحقق السريع من روابط الوثائق والمسارات المشار إليها فيها:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/check-markdown.ps1
```

أو مباشرة عبر CTest:

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build -j
cd build
ctest --output-on-failure
```

## الاستخدام

الشكل المقصود لسطر الأوامر:

```bash
nazm [options] <source.مجمع>
```

أهم الخيارات:

```text
-o <output>       تحديد مسار ملف الخرج
-f elf64|coff     اختيار صيغة الملف الكائني
-v                عرض رسائل تفصيلية
--version         عرض الإصدار وهدف بناء المشغل
--help            عرض المساعدة
```

مثال:

```bash
./build/nazm examples/مرحبا.مجمع -o build/مرحبا.o -f elf64
```

بعد إنتاج ملف كائني صالح يمكن فحصه بأدوات مثل `readelf`/`objdump` أو ربطه بأداة ربط النظام المناسبة.

مثال لاختيار COFF:

```bash
./build/nazm examples/مرحبا.مجمع -o build/مرحبا.obj -f coff
```


## التشخيصات العربية

يعرض نَظْم الأخطاء بالعربية مع موضع المصدر عند توفر النص الأصلي:

```text
خطأ في examples/diagnostics/فاصلة-مفقودة.مجمع:1:9: توقعت فاصلة عربية '،' بين المعاملات، لكنني وجدت: REGISTER
  السطر │ احمل ر0 ر1
        │         ^^ هنا
```

توجد أمثلة مقصودة الخطأ في [`examples/diagnostics/`](examples/diagnostics/) لاختبار شكل التشخيصات يدويا.

## مبادئ التطوير

- العربية أولا في واجهة المستخدم، أسماء أمثلة التجميع، والتشخيصات.
- صحة الترميز أهم من كثرة التعليمات: كل تعليمة يجب أن تنتج بايتات صحيحة ومختبرة.
- الذاكرة مملوكة بوضوح: معظم بيانات خط الأنابيب تُدار عبر `Arena`.
- لا توجد اعتماديات تشغيل خارجية.
- يفضل التغيير الصغير القابل للاختبار على إعادة الهيكلة الواسعة.
- عند إضافة تعليمة جديدة، يجب تحديث جدول الكلمات المفتاحية، جدول الترميز، الاختبارات، والوثائق.
