# نَظْم

**نَظْم** هو مُجمِّع عربي لبنية **x86-64** مكتوب بلغة C11. هدفه تحويل ملفات تجميع عربية بامتداد `.مجمع` إلى ملفات كائنية قياسية يمكن ربطها بأدوات النظام مثل `ld` على لينكس أو أدوات COFF على ويندوز.

المشروع في مرحلة مبكرة `0.1.0`: توجد بنية خط أنابيب واضحة، ووحدات أساسية للترميز، والرموز، والمحلل المعجمي، وجدول الرموز، ومخرجات ELF64/COFF. بعض الأجزاء ما زالت هيكلية أو محدودة، لذلك راجع [ROADMAP.md](ROADMAP.md) لمعرفة الأولويات الدقيقة.

## الفكرة

نَظْم يحاول جعل البرمجة منخفضة المستوى أقرب للناطقين بالعربية دون التخلي عن صرامة تنسيق الملفات الكائنية أو قواعد x86-64. يقرأ المصدر العربي كـ UTF-8، يتعامل مع التعليمات والسجلات والوسوم العربية، ثم يمررها عبر خط أنابيب متعدد المراحل:

```text
مصدر .مجمع
  -> التحليل المعجمي
  -> التحليل النحوي
  -> المرور الأول: أحجام التعليمات والوسوم
  -> المرور الثاني: ترميز x86-64
  -> كاتب المخرجات: ELF64 أو COFF
  -> ملف .o أو .obj
```

## الحالة الحالية

- الإصدار الحالي في `CMakeLists.txt` وواجهة `include/nazm.h` هو `0.1.0`.
- اللغة الأساسية للمشروع هي C11 بدون اعتماديات تشغيل خارجية.
- الاختبارات الحالية تغطي وحدات الذاكرة، Unicode العربي، جدول الرموز، الكلمات المفتاحية، الفوريّات، بادئة REX، المحلل المعجمي، المحلل النحوي، الترميز، المرورين، ELF64، وتحليل خيارات CLI.
- يرفض مشغل CLI ملفات المصدر الأكبر من 100 MiB برسالة عربية واضحة بدلا من محاولة تحميلها كاملة.
- اختبارات التكامل الكاملة مخطط لها ولم تكتمل بعد.
- إخراج ELF64 وCOFF موجود في الشجرة، لكن الطريق إلى مخرجات كائنية مكتملة ومتحقق منها ما زال ضمن خارطة الطريق.

## المتطلبات

- CMake 3.20 أو أحدث.
- مترجم C يدعم C11، مثل GCC 11+ أو Clang 14+.
- على لينكس: `make` ويفضل توفر `ld` أو `lld` للتجارب النهائية.
- على ويندوز: MinGW أو MSVC، مع مراعاة أن دعم COFF يحتاج تحقق مستمر.

## البناء

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build -j
```

أو عبر المسار المباشر الخفيف دون CMake:

```bash
./build.sh
```

## الاختبار

```bash
./build.sh test
```

يشغل المسار المباشر حاليا 12 مجموعة اختبارات وحدة. ويمكن تشغيل المجموعة نفسها عبر CTest بعد بناء CMake.

وللتحقق السريع من روابط الوثائق والمسارات المشار إليها فيها:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/check-markdown.ps1
```

أو مباشرة عبر CTest:

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build -j
cd build
ctest --output-on-failure
```

## الاستخدام

الشكل المقصود لسطر الأوامر:

```bash
nazm [options] <source.مجمع>
```

أهم الخيارات:

```text
-o <output>       تحديد مسار ملف الخرج
-f elf64|coff     اختيار صيغة الملف الكائني
-v                عرض رسائل تفصيلية
--version         عرض الإصدار
--help            عرض المساعدة
```

مثال:

```bash
./build/nazm examples/1.مجمع -o build/1.o -f elf64
```

بعد إنتاج ملف كائني صالح يمكن ربطه بأداة ربط النظام المناسبة.

## بنية المشروع

```text
src/
  alloc/       مخصص ذاكرة arena
  unicode/     أدوات UTF-8 والحروف العربية
  error/       تمثيل الأخطاء وطباعتها
  lexer/       التحليل المعجمي والكلمات المفتاحية العربية
  parser/      بناء قائمة التعليمات والمعاملات
  symtable/    جدول الوسوم والرموز
  passes/      المرور الأول والثاني
  encoder/     ترميز x86-64: REX وModRM والفوريّات وجدول التعليمات
  output/      كتاب مخرجات ELF64 وCOFF
  cli/         تحليل خيارات سطر الأوامر
  main.c       مشغل خط الأنابيب

include/
  nazm.h       واجهة تضمين مستقبلية للأدوات الخارجية مثل باء

tests/
  unit/        اختبارات الوحدة
  vendor/      نسخة Unity المستخدمة في الاختبارات

Docs/
  ARCHITECTURE.md
  STRUCTURE.md
  STACK.md
  CONVENTIONS.md
  TESTING.md
  INTEGRATIONS.md
  CONCERNS.md
```

## مبادئ التطوير

- العربية أولا في واجهة المستخدم، أسماء أمثلة التجميع، والتشخيصات.
- صحة الترميز أهم من كثرة التعليمات: كل تعليمة يجب أن تنتج بايتات صحيحة ومختبرة.
- الذاكرة مملوكة بوضوح: معظم بيانات خط الأنابيب تُدار عبر `Arena`.
- لا توجد اعتماديات تشغيل خارجية.
- يفضل التغيير الصغير القابل للاختبار على إعادة الهيكلة الواسعة.
- عند إضافة تعليمة جديدة، يجب تحديث جدول الكلمات المفتاحية، جدول الترميز، الاختبارات، والوثائق.
