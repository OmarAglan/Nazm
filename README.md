# نَظْم

**نَظْم** هو مجمع عربي لبنية **إكس86-64** (`x86-64`) مكتوب بلغة C11.
هدفه تحويل ملفات تجميع عربية بامتداد `.نظم` إلى ملفات كائنية قياسية يمكن
ربطها بأدوات النظام مثل `ld` على لينكس أو أدوات كوف (`COFF`) على ويندوز.

المشروع في مرحلة مبكرة `0.4.0`: توجد بنية خط أنابيب واضحة، ووحدات أساسية
للترميز والرموز، ومخرجات إلف64 (`ELF64`) وكوف. يدعم الإصدار الحالي أقسام
`.text` و`.data` وسلاسل UTF-8 العربية وترحيلات أساسية لمراجع الوسوم داخل
`.text`، مع بقاء التحقق النهائي ضد أدوات الربط على ويندوز ضمن خارطة الطريق.

الهدف الاستراتيجي هو أن يصبح نَظْم المجمع الإنتاجي لمصرف **باء**:
يستبدل خرج AT&T وجاس (`GAS`) الإنجليزي الحالي بمصدر تجميع عربي قابل للفحص،
ثم ينتج ملفات إلف64 وكوف قبل تمريرها إلى رابط النظام. وعلى المدى البعيد، بعد تثبيت
العقد واختبارات المطابقة، يُعاد تنفيذ نَظْم بلغة باء ويُبنى عبر مسار bootstrap
يحافظ على تنفيذ C11 الحالي كمرحلة صفر موثوقة.

> **تنبيه الحالة:** نَظْم ليس جاهزا بعد ليحل محل جاس في بناء باء الإنتاجي.
> أُغلقت بوابة صحة الترميز الحالية 7.1، لكن ما زالت توجد فجوات في الرموز
> الخارجية والترحيلات وتغطية تعليمات باء والتحقق الفعلي عبر روابط لينكس
> وويندوز.

## الفكرة

نَظْم يحاول جعل البرمجة منخفضة المستوى أقرب للناطقين بالعربية دون التخلي عن
صرامة تنسيق الملفات الكائنية أو قواعد إكس86-64. يقرأ المصدر العربي كـ UTF-8،
ثم يمرره عبر خط أنابيب متعدد المراحل:

```text
مصدر .نظم
  -> التحليل المعجمي
  -> التحليل النحوي
  -> المرور الأول: أحجام التعليمات والوسوم
  -> المرور الثاني: ترميز إكس86-64
  -> كاتب المخرجات: إلف64 أو كوف
  -> ملف .o أو .obj
```

## الحالة الحالية

- الإصدار الحالي في `CMakeLists.txt` وواجهة `include/nazm.h` وأمر `--إصدار`
  هو `0.4.0`. يعرض الأمر الهدف العربي ثم المعرف الدقيق مثل
  `إكس86-64-لينكس؛ x86_64-linux`.
- اللغة الأساسية للمشروع هي C11 بدون اعتماديات تشغيل خارجية.
- الاختبارات الحالية تغطي وحدات الذاكرة، Unicode العربي، جدول الرموز، الكلمات المفتاحية، الفوريّات، بادئة REX، المحلل المعجمي، المحلل النحوي، الترميز، المرورين، ELF64، COFF، التشخيصات، أمثلة المصدر، وتحليل خيارات CLI.
- يرفض مشغل CLI ملفات المصدر الأكبر من 100 MiB برسالة عربية واضحة بدلا من محاولة تحميلها كاملة.
- تعرض التشخيصات العربية الآن موضع الخطأ مع السطر الأصلي وسهم يحدد امتداد الرمز أو المعامل عند توفر المصدر.
- يرفض المحلل النحوي إزاحات الذاكرة خارج مجال 32 بت الموقع، ويرفض المرور الثاني أي قفز
  أو نداء نسبي لا تمثله `rel32` بدلا من اقتطاع القيمة.
- يطابق الإصدار 0.4 الكلمات المفتاحية والوسوم كبايتات UTF-8 دون تطبيع أو
  أسماء بديلة ضمنية؛ أسماء المصدر المعتمدة بلا حركات. راجع
  [سياسة Unicode](Docs/UNICODE.md).
- الوسوم محلية افتراضيا؛ يصدّر `.عام اسم_الوسم` الرمز للرابط، ويثبّت
  `.محلي اسم_الوسم` الرؤية المحلية صراحة. يقبل التصريح قبل تعريف الوسم أو
  بعده، ويرفض التعارض أو بقاء الوسم بلا تعريف.
- تغطي اختبارات الوحدة الأمثلة الأربعة المسماة بالعربية داخل الذاكرة، ويجمع
  اختبار قبول مستقل كل ملفات `examples/*.نظم` المشحونة إلى إلف64 وكوف، مستخدما
  المشغلين `nazm` و`نظم`.
- يثبت [عقد القدرات المولد](Docs/generated/nazm_capabilities_v1.json) التعليمات
  وأشكال المعاملات والأحجام والتوجيهات والأقسام والرموز والترحيلات المنفذة،
  وتجمع بوابة قبول مستقلة fixtures أشكال باء المدعومة إلى إلف64 وكوف.
- إخراج إلف64 وكوف يدعم `.text` و`.data` وجداول الرموز وترحيلا أساسيا من
  `.text` إلى الوسوم، لكن الربط النهائي ما زال يحتاج تحقق CI ومنصات فعلية.
- عقد باء النهائي يستخدم كتلة `نظم { ... }` فقط ويحذف مسار جاس الخام. تنفيذ
  التغيير داخل مستودع باء مهمة مستقلة بعد اكتمال تغطية تعليماته. التفاصيل في
  [Docs/BAA_INTEGRATION.md](Docs/BAA_INTEGRATION.md).

## المتطلبات

- CMake 3.20 أو أحدث.
- مترجم C يدعم C11، مثل GCC 11+ أو Clang 14+.
- على لينكس: `make` ويفضل توفر `ld` أو `lld` للتجارب النهائية.
- على ويندوز: MinGW أو MSVC، مع مراعاة أن دعم COFF يحتاج تحقق مستمر.

## البناء

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build -j
```

أو عبر المسار المباشر الخفيف دون CMake:

```bash
./build.sh
```

## الاختبار

```bash
./build.sh test
```

يشغل المسار المباشر حاليا 18 مجموعة اختبارات. تشغل CTest المجموعات نفسها،
وتضيف اختباري قبول لواجهة سطر الأوامر وكل الأمثلة، ومقارنة بايتات تفاضلية مع
جنو (`GNU`) `as` و`objcopy` عندما تتوفر الأداتان.

وللتحقق السريع من روابط الوثائق والمسارات المشار إليها فيها:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/check-markdown.ps1
```

أو مباشرة عبر CTest:

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build -j
cd build
ctest --output-on-failure
```

## الاستخدام

الشكل المقصود لسطر الأوامر:

```bash
نظم [خيارات] <ملف.نظم>
```

يبني CMake الأمر العربي `نظم` والأمر المحمول `nazm` من الملف التنفيذي نفسه.
تعرض الوثائق الأمر العربي أولا، ويبقى `nazm` صالحا لبيئات البناء والحزم.

أهم الخيارات:

```text
-خ، --خرج <ملف>       تحديد مسار الملف الكائني
-ك، --كشف <ملف>       كتابة كشف التجميع
-ص، --صيغة <صيغة>     اختيار إلف64 أو كوف
-ت، --تفصيل           عرض رسائل تفصيلية
--إصدار               عرض الإصدار وهدف البناء
-م، --مساعدة          عرض المساعدة
```

مثال:

```bash
./build/نظم examples/مرحبا.نظم --خرج build/مرحبا.o --صيغة إلف64
```

بعد إنتاج ملف كائني صالح يمكن فحصه بأدوات مثل `readelf`/`objdump` أو ربطه بأداة ربط النظام المناسبة.

لإنتاج كشف تجميع نصي UTF-8 بجانب الملف الكائني:

```bash
./build/نظم examples/مرحبا.نظم --خرج build/مرحبا.o \
  --كشف build/مرحبا.كشف --صيغة إلف64
```

يسجل الملف رقم سطر كل عبارة مصدرية وقسمها وإزاحتها النسبية والبايتات
النهائية التي أصدرها المرور الثاني. تُلف بيانات السطر الطويلة كل 16 بايتا،
ولا يعيد كشف التجميع ترميز التعليمات أو تخمين أحجامها. يجب أن تختلف مسارات
المصدر والملف الكائني والكشف حتى لا يكتب أحدها فوق الآخر؛ يتحقق المشغل من هوية
المسارات المطبّعة، لا من تطابق النص وحده.

مثال لاختيار COFF:

```bash
./build/نظم examples/مرحبا.نظم --خرج build/مرحبا.obj --صيغة كوف
```


## التشخيصات العربية

يعرض نَظْم الأخطاء بالعربية مع موضع المصدر عند توفر النص الأصلي:

```text
خطأ في examples/diagnostics/فاصلة-مفقودة.نظم:1:17: توقعت فاصلة عربية '،' بين المعاملات، لكنني وجدت: سجل
  السطر │ انقل سجل_المركم سجل_العداد
        │                ^^^^^^^^^^ هنا
```

توجد أمثلة مقصودة الخطأ في [`examples/diagnostics/`](examples/diagnostics/) لاختبار شكل التشخيصات يدويا.

## مبادئ التطوير

- العربية أولا في واجهة المستخدم، أسماء أمثلة التجميع، والتشخيصات.
- صحة الترميز أهم من كثرة التعليمات: كل تعليمة يجب أن تنتج بايتات صحيحة ومختبرة.
- الذاكرة مملوكة بوضوح: معظم بيانات خط الأنابيب تُدار عبر `Arena`.
- لا توجد اعتماديات تشغيل خارجية.
- يفضل التغيير الصغير القابل للاختبار على إعادة الهيكلة الواسعة.
- عند إضافة تعليمة جديدة، يجب تحديث جدول الكلمات المفتاحية، جدول الترميز، الاختبارات، والوثائق.
